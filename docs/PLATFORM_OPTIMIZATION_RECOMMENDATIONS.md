# 智能投标预审智能体 — 整体优化建议（评审稿）

> 本文档为产品与工程视角的改进建议汇总，**不要求立即改代码**，可按优先级分阶段采纳。  
> 基于当前仓库结构（Next.js 15、NextAuth v5、Drizzle、PostgreSQL、MinerU/AI 相关能力）及常见落地问题整理。

---

## 一、文档与品牌一致性

| 问题 | 建议 |
|------|------|
| 根目录 `README.md` 已统一为「智能投标预审智能体」业务描述 | 保持 README 标题、简介与实际仓库一致，便于新成员与对外协作 |
| `docs/ROADMAP.md` 中多项（迁移、OAuth、密码重置、中间件等）与现状可能已部分重叠 | 定期做一次 **ROADMAP 与代码对照**，勾选已完成项、拆分「规划中」与「已弃用」，避免执行层判断失真 |

---

## 二、认证、安全与合规

| 方向 | 建议 |
|------|------|
| **路由保护** | 评估是否引入 `middleware` 对 `(dashboard)` 下路由做会话校验，与页面内 `auth()` 形成纵深；注意与静态资源、NextAuth 回调路径的排除规则 |
| **忘记密码** | 生产环境关闭或严格限制 `_devResetUrl` 类调试信息；对 `POST /api/auth/forgot-password` 做 **按 IP/邮箱的限流**，防枚举与刷库 |
| **密码与令牌** | 重置链接一次性、短时效已在设计内；可补充：成功后使同用户其他会话失效（按需）、审计日志记录重置行为 |
| **OAuth 与本地密码** | 明确「仅 OAuth」「仅密码」「两者兼有」的产品规则，并在登录/注册/重置文案中写清，减少客服类问题 |
| **数据敏感度** | 招标文件、投标文件多属商业秘密：在部署文档中明确 **访问控制、备份加密、日志脱敏、保留周期** 与责任人 |

---

## 三、数据库与 Schema 演进

| 方向 | 建议 |
|------|------|
| **`drizzle-kit push`** | 在部分 PostgreSQL 版本上曾出现误生成大量 `DROP …_not_null` 的问题；生产环境优先 **`generate` + 人工审核 SQL** 或固定迁移流水线，避免一次 push 误伤全库 |
| **新表落地** | 对关键增量（如 `password_reset_tokens`）保留 **可独立执行的 SQL 脚本** 作为应急通道（仓库内已有示例思路时可纳入运维手册） |
| **备份与恢复** | 文档化：备份频率、恢复演练周期、与 MinerU 解析产物（上传目录）是否一并纳入备份范围 |

---

## 四、配置、部署与运行环境

| 方向 | 建议 |
|------|------|
| **`AUTH_URL` / `NEXT_PUBLIC_APP_URL`** | 局域网或域名访问时，与 Cookie、OAuth 回调、邮件内链接强相关；在《部署检查清单》中列为 **必核项** |
| **`DATABASE_URL`** | Windows 下 `localhost` 与 IPv6/`127.0.0.1` 行为差异可写入 FAQ；生产连接串避免提交到仓库 |
| **环境变量** | 维护「必填 / 可选 / 仅开发」三组列表（含 MinerU、Redis、LangChain、Resend 等），并在启动时做弱校验（将来可做，本文仅建议） |
| **`.next` 与构建产物** | 明确不提交 `.next`；CI 中每次干净构建，减少「缺 routes-manifest」类环境问题 |

## 五、可观测性与运维

| 方向 | 建议 |
|------|------|
| **结构化日志** | 对上传、解析、报告生成等长链路打 **requestId / documentId / projectId**，便于排障 |
| **错误暴露** | API 对客户端返回统一错误码与简短中文提示，详细堆栈仅服务端日志 |
| **健康检查** | 对外部署时提供只读 `health`（数据库、磁盘、MinerU 可达性可选），便于负载均衡与告警 |

---

## 六、性能与体验（产品向）

| 方向 | 建议 |
|------|------|
| **大文件与解析** | 明确单文件大小上限、超时策略、失败重试与用户对「解析中」状态的预期（文案 + 轮询/推送长期方案） |
| **批量操作** | 文档管理已支持批量解析/删除等方向；建议在帮助或首次引导中说明 **与 MinerU 并发能力的关系**，避免一次选中过多导致排队或失败 |
| **导航信息架构** | 「文件管理」全局列表 vs 「项目文档管理」能力差异，建议在 UI 文案或轻量帮助中写一句，降低误点后的困惑 |
| **无障碍与国际化** | 若面向政企用户，可规划键盘操作、读屏标签、简繁/中英文切换的阶段性目标 |

---

## 七、工程与质量保障

| 方向 | 建议 |
|------|------|
| **CI** | 至少包含：`lint`、`typecheck`、（可选）`build`；数据库迁移类变更单独流水线或人工 gate |
| **依赖与安全** | 定期 `npm audit`；关注 Next.js / next-auth 大版本说明 |
| **占位页面** | 设置、统计分析等占位区，可在 ROADMAP 中标为「MVP 外」或排期模块，避免用户以为功能故障 |

---

## 八、建议优先级（摘要）

1. **P0（安全与数据）**：生产密钥与链接、忘记密码限流、备份与权限说明文档化。  
2. **P1（稳定性）**：数据库迁移策略从「仅靠 push」转向「可审迁移」；部署检查清单（`AUTH_URL`、DB、MinerU）。  
3. **P2（体验与品牌）**：README/ROADMAP 与产品对齐；全局 vs 项目文档路径的用户说明。  
4. **P3（长期）**：可观测性、健康检查、国际化与无障碍。

---

## 九、结语

当前技术栈适合继续迭代「文档上传 → 解析 → 审查 → 报告」闭环。上述项多为 **流程、文档与发布纪律** 上的加固；具体实施时可按季度复盘，与本文件一并更新版本与日期。

*文档版本：建议随仓库以 Git 管理；首次撰写为评审稿，可在评审会后更新「已采纳 / 暂缓 / 拒绝」列。*
