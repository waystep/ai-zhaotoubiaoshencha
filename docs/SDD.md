# 智能投标预审平台 SDD 体系

> SDD = Specification Driven Development，规范驱动开发。  
> 目标：让 2.0 之后的每次迭代，都先形成可评审、可测试、可追溯的规范，再进入实现。

## 1. 为什么需要 SDD

智能投标预审平台不是普通 CRUD 系统。它涉及业务规则、文档证据、AI 判断、人工复核、组织权限和报告责任边界。如果直接从页面或模型提示词开始开发，很容易出现三类问题：

- AI 输出看似合理，但无法追溯到业务规则和文档证据。
- 功能能演示，但缺少验收口径，后续难以判断是否变好。
- 代码持续增加，但产品方法论和审查经验没有沉淀。

SDD 的作用是把“为什么做、做给谁、怎么验收、有哪些边界”写在代码之前，并让规范持续约束实现。

## 2. 规范分层

| 层级 | 文件建议 | 解决的问题 | 负责人 |
|---|---|---|---|
| 产品意图 | `Why.md` | 为什么值得做，用户是谁，场景如何验证 | 产品/业务 |
| 版本目标 | `docs/roadmap/VERSION.md` | 本版本解决哪些问题，不解决哪些问题 | 产品/技术 |
| 业务规范 | `docs/specs/*.md` | 角色、场景、规则、状态机、异常口径 | 产品/业务/技术 |
| 技术设计 | `docs/architecture/*.md`、`DESIGN.md` | 架构、数据模型、接口、任务流 | 技术 |
| 验收规范 | `docs/acceptance/*.md` | 用例、测试数据、验收清单、风险 | 产品/测试 |
| 运行规范 | `docs/deployment/*`、`docs/operations/*` | 部署、环境变量、监控、回滚 | 技术/运维 |

当前 1.0 已有 `Why.md`、`DESIGN.md`、`docs/项目方案说明书.md`、`docs/2026-CICC-设计文档-RFC.md` 和专题技术文档。2.0 开始建议补齐 `docs/specs/` 与 `docs/acceptance/` 两类目录。

每个 2.0 需求进入开发前，至少补一份规范。建议模板如下：

```markdown

- 当前痛点：
- 为什么现在做：

## 目标
- 本次要达成：
- 明确不做：

## 用户故事
- 作为...我希望...以便...

## 业务规则
- 规则 1：
- 规则 2：

## 数据与状态
- 涉及实体：
- 状态变化：
- 审计/权限：

## AI 行为规范
- 输入：
- 输出结构：
- 证据要求：
- 禁止行为：
- 失败兜底：

## 页面与交互
- 入口：
- 主流程：
- 空状态/错误态：

## 接口与存储
- API：
- 表结构：
- 迁移策略：

## 验收标准
- Given / When / Then：
- 必测用例：
- 手工验收清单：

## 风险
- 业务风险：
- 技术风险：
- 数据风险：
```

## 4. 2.0 迭代工作流

1. 先更新 `Why.md` 或对应版本目标，确认需求是否服务核心用户。
2. 为每个功能写 `docs/specs/<feature>.md`，明确范围、规则、AI 边界和验收标准。
3. 修改 `DESIGN.md` 或专题架构文档，确保数据模型、API、Agent、任务流有设计依据。
4. 先写或补测试：权限、API、状态机、关键 UI 流程、Agent 输出结构。
5. 实现代码，尽量沿用现有 Next.js、Drizzle、Mastra、shadcn/ui 风格。
6. 运行 `npm run build`，必要时补充脚本级验证。
7. 更新验收文档与部署说明，记录已知限制。

## 5. AI 审查类需求的特殊规范

涉及审查、评分、风险判断、报告生成的需求，必须额外写清楚：

| 项 | 要求 |
|---|---|
| 证据 | 输出应尽量包含页码、区块、bbox 或原文摘录，不允许只有结论 |
| 置信度 | 需要区分确定问题、疑似问题、需人工复核 |
| 禁止项 | 不输出替代法律意见，不伪造未在材料中出现的事实 |
| 可回放 | Agent 输入、工具调用、落库结果应能按报告或任务追踪 |
| 人工反馈 | 2.0 后建议支持确认、驳回、修改和备注，用于后续数据验证 |

## 6. Definition of Done

一个需求只有同时满足以下条件，才算完成：

- 有对应规范文档，且范围、边界、验收标准明确。
- 代码实现与规范一致，没有隐藏的业务规则。
- 涉及权限、组织隔离或数据写入的 API 有测试或手工验证记录。
- AI 输出有结构化结果和证据回链。
- `npm run build` 通过。
- README、部署、运维或用户说明在必要时同步更新。

## 7. 可复用 Skill

本仓库已提供一个项目级 skill 草案：`skills/tender-review-sdd/SKILL.md`。后续在 Codex 中迭代本平台时，可以把它安装到本地 skills 目录，或直接让智能体读取该文件作为项目工作流规范。
